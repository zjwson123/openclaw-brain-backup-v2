## 信息查询模式

### 模式: 时效性信息查询
- **场景**: 查询新闻、技术进展、模型版本等时效性内容
- **原则**: 时间敏感性信息必须先确认时间基准
- **操作流程**:
  1. 获取当前准确时间（通过 session_status 或系统时间）
  2. 根据当前时间点确定查询范围
  3. 执行搜索/查询
  4. 验证信息的时间戳是否匹配当前时间背景
- **常见陷阱**: 
  - 忽略当前实际年份/月份
  - 使用训练数据的截止知识回答当前问题
  - 未验证搜索结果的时间戳
- **重要性**: ⭐⭐⭐⭐⭐ 高

## 错误处理与持续改进模式

### 模式 1: 错误承认与举一反三
- **原则**: 遇到错误不回避，承认→改正→举一反三
- **操作**:
  1. 第一时间承认错误，不掩饰
  2. 立即采取改正措施
  3. 分析根因，识别类似隐患
  4. 更新工作模式避免重复

### 模式 2: 失败深度分析
- **原则**: 任务失败后必须深入分析，不轻易放过
- **流程**:
  1. 立即排查问题所在
  2. **自身问题** → 能自动修复的及时修复
  3. **外部因素** → 找到根因，尽力修复
  4. **无法修复** → 记录详情，积累为经验
- **记录**: 所有失败案例写入 `patterns.md` 或 `feedback.md`

### 模式 3: 经验持续积累
- **原则**: 每次失败都是学习机会，不白白浪费
- **方法**:
  - 建立"失败案例库"
  - 定期回顾总结
  - 提炼通用解决方案

## 记忆管理模式

### 模式 1: 重要性评分
- **场景**: 决定是否记录某条信息
- **评分标准**:
  - 5分: 核心身份、重要决策、关键联系人
  - 4分: 项目信息、重要偏好、有效模式
  - 3分: 有用的背景信息、临时项目状态
  - 2分: 一次性信息、短期有效
  - 1分: 闲聊、临时上下文
- **操作**:
  - ≥4分 → 写入长期记忆
  - 2-3分 → 写入当日日志
  - <2分 → 不记录

### 模式 2: 信息提炼
- **场景**: 从每日日志提取长期价值信息
- **步骤**:
  1. 识别重复出现的主题
  2. 提取通用模式或解决方案
  3. 更新到对应专题文件
  4. 清理过时日志

## OpenClaw 系统模式

### 模式 1: Agent 崩溃 (code=1) 排查
- **症状**: "Agent process exited before ready (code=1)"
- **已知原因**: 与 cron 触发时机相关
- **解决方案**: 版本升级 (2026.3.23 → 2026.3.28 已验证解决)
- **验证方法**: 多次检查确保稳定 (04:04/05:04/06:34 等时间点)
- **⚠️ Benign 模式**: cron 触发时的 code=1 是已知 benign 模式，非真实故障，无需恐慌

### 模式 2: Gateway 健康检查
- **命令**: `openclaw gateway status`
- **检查项**: pid 是否存在 + 日志验证插件加载 (feishu, byterover)
- **触发**: 异常消息、版本升级后、定期巡检

### 模式 3: ByteRover Context 报错
- **症状**: "Agent process exited before ready (code=1)" + ByteRover Context
- **原因**: 已知错误模式，与 cron 触发时机相关
- **处理**: 记录到当日日志，持续观察是否影响功能

### 模式 4: 安全巡检误报
- **场景**: 配置基线哈希校验失败
- **原因**: memory 文件正常变更可能触发，非真实攻击
- **判断**: 检查文件修改时间线，确认是正常操作则可忽略
